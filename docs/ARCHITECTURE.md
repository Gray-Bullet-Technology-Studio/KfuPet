## 项目介绍

Kfu AI Pet（快芙桌宠）是一款基于 **WPF** 开发的桌面宠物应用。当前版本为原型阶段，专注于基础窗口系统与交互体验；未来将逐步引入 **混合动作系统**，结合本地预设动画与 AI 实时驱动，实现富有创意的角色交互表现。

### 当前已实现特性

### 记忆系统

记忆系统是 KfuPet 的核心，决定了 AI 是否具有"生命感"。它分为六层架构，而非简单的聊天记录数据库：

```
Memory
│
├── ShortMemory          // 短期记忆（工作记忆）
├── LongMemory           // 长期记忆（持久化）
├── UserProfile          // 用户画像
├── RelationshipMemory   // 关系记忆
├── ExperienceMemory     // 经历记忆
└── SelfMemory           // 自我记忆
```

#### 第一层：Short Memory（短期记忆）

AI 当前正在思考时使用的工作记忆，容量有限（建议 50-200 条），超过后自动淘汰，不永久保存。

**生命周期：**
```
收到消息 → 存入 ShortMemory → Brain 读取 → 判断重要性
                                          ├─ 是 → 转 LongMemory
                                          └─ 否 → 自动删除
```

#### 第二层：Long Memory（长期记忆）

用户偏好等持久化记忆，使用 SQLite 存储，具有权重系统。

**数据结构：**
| 字段 | 说明 | 示例 |
|------|------|------|
| ID | 唯一标识 | 自增整数 |
| Type | 类型 | Preference / Experience / Event |
| Content | 记忆内容 | 用户非常喜欢初音未来 |
| Weight | 权重（0-100） | 98（越高越重要） |
| CreateTime | 创建时间 | 2026-07-01 10:30:00 |
| LastAccess | 最后访问时间 | 2026-07-08 15:20:00 |
| Importance | 重要性等级 | High / Medium / Low |
| Tags | 标签 | 偏好、初音未来 |

**权重机制：** 低权重记忆（如"今天喝奶茶"，Weight 3）会在一段时间后删除；高权重记忆（如"主人生日"，Weight 100）永久保存。

#### 第三层：Relationship Memory（关系记忆）

记录与用户的关系维度，动态变化，影响说话方式。

```
Relationship
│
├── Trust          // 信任度
├── Familiarity    // 熟悉度
├── Respect        // 尊重度
├── Attachment     // 依恋度
├── Dependence     // 依赖度
└── InteractionCount // 互动次数
```

**示例：** 初识时说"您好"，一年后说"今天怎么这么晚才回来？"

#### 第四层：Experience Memory（经历记忆）

记录重要事件和里程碑，而非普通聊天。

```
2026 - 第一次聊天
      - 第一次一起过生日
      - 第一次联网
      - 第一次安装角色
```

这些经历赋予 AI"回忆"能力，增强生命感。

#### 第五层：Self Memory（自我记忆）

记录 AI 的自我成长经历，是 KfuPet 区别于其他桌宠的关键。

```
今天学会了一个新动作
今天第一次成功联网
今天主人夸奖了我
今天回答失败了
```

#### 第六层：User Profile（用户画像）

结构化的用户信息，用于个性化交互。

```
User
│
├── Name       // 姓名
├── Birthday   // 生日
├── Preference // 偏好
├── Language   // 语言
└── Habit      // 习惯
```

#### Memory Consolidation（记忆整理）

参考人类记忆形成过程，增加记忆整理器，避免 Memory 变成聊天日志：

```
用户输入 → Short Memory → Memory Consolidation
                              ├── 判断重要性
                              ├── 去除重复内容
                              ├── 更新已有记忆
                              ├── 提升权重
                              └── 提取用户偏好
                              → Long Memory
```

**示例：** 用户多次提到喜欢初音未来，整理器不会创建多条重复记忆，而是更新为：
```
类型：Preference
内容：用户非常喜欢初音未来
权重：98
最后更新时间：2026-07-08
证据次数：6
```

#### Memory Manager

所有 Memory 统一经过 MemoryManager，避免 Brain 直接访问 SQLite：

```
Brain → MemoryManager → SQLite
```

这样设计便于未来扩展云同步功能，无需修改 Brain 模块。

### 情绪系统

情绪由系统维护，而非让 AI 自己生成，确保情绪连续性和一致性。

#### Emotion Engine

管理多种情绪值，支持多值并存：

| 情绪 | 说明 |
|------|------|
| Happy | 开心 |
| Sad | 难过 |
| Curious | 好奇 |
| Sleepy | 困倦 |
| Lonely | 孤独 |
| Excited | 兴奋 |
| Angry | 生气 |
| Embarrassed | 尴尬 |

**示例：** Happy 80、Sleepy 20、Curious 50 可同时存在。

#### 情绪触发规则

```
聊天成功      → Happy +10
被主人夸奖    → Happy +20, Attachment +5
主人一天未打开 → Lonely +15
AI 回答失败   → Confidence -10
```

#### 情绪衰减

情绪值会随时间逐渐衰减，恢复到正常水平（基准值 50）：

衰减公式：`当前值 = 当前值 - 衰减系数 × 时间间隔(秒)`

| 情绪类型 | 衰减系数 |
|---------|---------|
| Happy | 0.02/s |
| Sad | 0.03/s |
| Angry | 0.05/s |
| Excited | 0.04/s |
| Curious | 0.03/s |
| Sleepy | 0.02/s |
| Lonely | 0.03/s |
| Embarrassed | 0.06/s |

```
Happy 80 → 78 → 76 → 74 → ... 恢复正常
```

避免 AI 永远处于极端情绪状态。

#### Personality（人格）

人格与情绪分离，人格不变，情绪变化：

```
Personality
│
├── 活泼
├── 高冷
├── 天然呆
└── 傲娇
```

**示例：** 傲娇人格即使 Happy 90，表现依然是傲娇风格。

#### 情绪对行为的影响

情绪直接影响 AI 的行为目标：

| 情绪 | 目标行为 |
|------|----------|
| Happy | 想聊天 |
| Sleepy | 想睡觉 |
| Curious | 观察鼠标 |

Brain 永远读取当前情绪状态来决策。

### 模块优先级

| 优先级 | 模块 | 说明 |
|--------|------|------|
| 1 | Memory Engine | 项目的灵魂，赋予 AI 生命感 |
| 2 | Emotion Engine | 让行为有连续性和个性 |

#### 核心设计原则

| 原则 | 说明 |
|------|------|
| **Renderer 抽象** | manifest 中写 `"Renderer": "Sprite"`，而非写死 WinUI。后续支持 Live2D/Spine/VRM 只需添加渲染器实现 |
| **配置分离** | manifest 管"这个角色是谁"，character 管"这个角色长什么样"，职责清晰，扩展方便 |
| **角色与 AI 解耦** | 角色不知道 AI，AI 只读取 Personality。一个角色可以切换不同人格，人格也可以跨角色复用 |
| **预留扩展点** | Skeleton、Personality、Prompt、Audio 等目录即使 v0.0.1 不用也预留，避免后期重构 |
| **AssetManager 统一管理** | 程序不直接加载图片，而是通过 `AssetManager.GetTexture("Head")` 获取，方便后续支持 WebP、Atlas、压缩包等格式 |

### 动作调度策略

系统采用 **预设优先** 的调度策略，以最小化 AI 调用成本：

```
动作请求 → 查询预设动作库 → 命中? → 是 → 直接播放预设动画
                              ↓ 否
                    调用 AI 生成动作序列
                              ↓
                    播放 AI 生成的动画
                              ↓
                    (可选) 沉淀为预设
```

- **基础动作**（打招呼、眨眼、伸懒腰等）直接调用预设动作库，零 AI 成本
- **AI 提议动作** 先查预设库，命中则复用，未命中才让 AI 自主生成
- **动作沉淀**：AI 生成的优质动作可保存到预设库，避免重复消耗 tokens

## 未来构想

### 构想一：跨平台桌宠互联生态

打造 **PC + Android + 开发板** 三端互联的桌宠社交网络。

- **设备互联**：当用户拥有多台安装桌宠的设备时，可选择是否开启互联。开启后，不同设备间的桌宠可以互相"串门"，在设备间自由穿梭。
- **用户互联**：用户之间可以互相添加好友，好友的桌宠之间也能实现互动与拜访。
- **陌生人随机访问**：用户可选择开启"随机串门"功能，系统会随机匹配其他在线用户，让你有机会遇到陌生人的桌宠，带来意外的惊喜与互动。
- **跨平台互联**：PC 桌宠可与 Android 端、开发板端的桌宠实现无缝互联，共享互动体验。

> **注**：Android 端和开发板端为独立项目。

### 构想二：开发板端二次元桌面时钟

基于 **ESP32-S3 等开发板** 开发独立运行的二次元桌面时钟硬件。

- **基础功能**：保留传统桌面时钟的时间显示，支持多种时钟界面风格。
- **融入桌宠**：在时钟界面中嵌入桌宠模块，让时钟不再单调，桌宠可陪伴你的每一次查看。
- **设备互联**：开发板时钟可与 PC 桌宠、Android 桌宠及其他开发板设备实现互联，共享桌宠互动体验。
- **通讯方式**：
  - **蓝牙 / WiFi**：近距离设备间直连，低延迟、高私密。
  - **互联网**：远距离跨网段通信，随时随地远程互联。

### 互联体系总览
